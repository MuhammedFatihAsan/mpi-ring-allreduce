Ring-Allreduce for Distributed Gradient Synchronization

Final Presentation — May 2026

MotivationThe synchronization bottleneck

Distributed training computes gradients on many workers and must aggregate them every iteration. The straightforward "send everything to one master" pattern stops scaling once the master saturates.

Centralized reduce-broadcast

Why it stalls

  • Bytes through the master grow as n · M.
  • Master's link is fixed bandwidth — it caps the whole job.
  • Adding workers increases the synchronization cost.
Goal. Replace this pattern with one whose per-rank traffic stays bounded as n grows.

IdeaArrange workers in a ring

Each worker talks only to its left and right neighbors. Data is partitioned and circulated; partial sums accumulate as chunks travel around. No single node ever sees all the traffic.

Ring topology, n = 5

Two phases

  • Reduce-scatter. Each rank sends one chunk per step and adds the incoming chunk into its local buffer. After n − 1 steps each rank owns the global sum of one chunk.
  • Allgather. Same neighbor pattern, but received chunks overwrite instead of summing. After another n − 1 steps every rank holds the full sum.
Bandwidth claim. Each rank moves about 2(n−1)/n · M bytes — bounded by 2M regardless of n.

How it worksReduce-scatter with n = 4

Each row is a rank, each cell is a chunk. Shaded cells are the chunk being updated that iteration. After n − 1 iterations, every rank owns the complete sum of exactly one chunk.

Allgather then runs the same neighbor pattern for another n − 1 steps, propagating fully-summed chunks around the ring without further addition.

ImplementationSendrecv-based loop in C

Each step performs a paired send-and-receive in one MPI call. MPI_Sendrecv guarantees no deadlock when every rank is sending to its right neighbor at the same time.

Reduce-scatter — src/ring_allreduce.c

for (int step = 0; step < size - 1; step++) {
    int send_chunk = (rank - step + size) % size;
    int recv_chunk = (rank - step - 1 + size) % size;

    // chunk-layout helpers handle ragged divisions
    size_t ss = get_chunk_start(count, size, send_chunk);
    size_t sc = get_chunk_size (count, size, send_chunk);
    size_t rs = get_chunk_start(count, size, recv_chunk);
    size_t rc = get_chunk_size (count, size, recv_chunk);

    MPI_Sendrecv(data + ss, sc, MPI_FLOAT, right, 0,
                 temp_buffer, rc, MPI_FLOAT, left,  0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);

    for (size_t i = 0; i < rc; i++)
        data[rs + i] += temp_buffer[i];
}

Design notes

  • Ragged divisions. When M % n ≠ 0, the leading ranks each carry one extra element. Helpers compute chunk start and size from any rank consistently.
  • Deadlock-free. Sendrecv matches one send to one receive atomically — no need to interleave non-blocking calls.
  • Two timings. Communication-only (between barriers around the loops) and setup+communication (includes neighbor-index computation and the temp buffer allocation).
  • Verification. All ranks initialize the array to 1.0; the expected post-reduction value is exactly n.

Profilinggprof confirms it's communication-bound

A separate make profile build (compiled with -O0 -pg so helpers don't get inlined away) generates gmon.out. gprof then attributes time to user-level symbols.

What we measured

  • Run on the largest sweep size so the 10 ms sampler has work to attribute.
  • User-level helpers (get_chunk_size, get_chunk_start, verify_results, …) account for a negligible share of total time.
  • Almost all wall-clock time is consumed inside MPI library calls, where gprof does not sample.
Takeaway. The application is communication-bound. Optimization effort would have to target the MPI implementation or the transport, not the user loop bodies.

ConclusionWhat the four parts tell us, together

Correctness. Every admitted run passes verification, including ragged divisions where M % n ≠ 0.
Ring wins everywhere. Across the swept range (256 KB → 100 MB) and every rank count from 2 to 16, ring is faster than the baseline. Speedups range from 1.26× to 3.71×.
Bandwidth claim, observed. Per-rank bytes for the ring stay bounded; the baseline's per-rank bytes scale with n. The measured curves match the predicted shape.
Limitation. Single-machine, shared-memory transport. The qualitative scaling is preserved but absolute throughput numbers do not represent a real cluster.
Bottom line. Ring-Allreduce delivers the predicted asymptotic advantage on a hobby-grade machine. The same code, recompiled against a real interconnect, is what production frameworks (Horovod, NCCL) ship today.
